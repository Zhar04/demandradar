# DemandRadar — план разработки

> Живой документ. Обновляется в начале и конце каждого этапа.
> Статусы этапов и score — в [PROGRESS.md](PROGRESS.md). Решения — в [DECISIONS.md](DECISIONS.md).

## 1. Что строим (одним абзацем)

Локальный (готовый к облаку) агрегатор сигналов спроса на наши товарные группы по КЗ/СНГ:
коннекторы собирают формализованный (тендеры), латентный (поисковый/маркетплейсы) и
опережающий (новые компании, стройки, новости) спрос, ядро нормализует всё в единую модель
`DemandSignal`, фильтрует по ключевым словам/кодам ТРУ, дедуплицирует, обогащает контактом ЛПР,
скорит и доставляет в Telegram + веб-дашборд с воронкой. ИИ опционален (Ollama/Claude Code),
по умолчанию выключен. Никаких платных API.

## 2. Критика и улучшения архитектуры из брифа (Этап 0)

Принято без изменений: коннекторы + единое ядро; единый сетевой слой с ProxyProvider;
LLMProvider с Null по умолчанию; SQLite с абстракцией; mock→live по наличию ключа в .env.

Изменения/уточнения (обоснование в DECISIONS.md):

1. **Синхронное ядро, а не asyncio.** Источников десятки, а не тысячи; частота опроса — минуты.
   Синхронный код с последовательным обходом коннекторов проще отлаживать и надёжнее 24/7.
   Параллелизм добавим на Этапе 7 пулом потоков на уровне «один поток = один коннектор», это
   не потребует переписывания коннекторов (они и так изолированы).
2. **Пайплайн — это функция, а не фреймворк.** collect → normalize → filter → dedup → classify →
   enrich → score → route реализуем как явную последовательность вызовов в `core/pipeline.py`.
   Без шин сообщений, очередей и плагинных систем (YAGNI). Расширение = новый коннектор.
3. **Классификация ДО обогащения и скоринга** (в брифе: дедуп → классификация → обогащение →
   скоринг — сохраняем), но фильтрация релевантности — ЧАСТЬ нормализации коннектора goszakup
   уже на Этапе 1 (по Lots.nameRu/descriptionRu + enstruList), чтобы не таскать в БД мусор.
   Нерелевантные лоты отбрасываются до записи, счётчик отбраковки — в лог/метрики.
4. **REST-фолбэк для goszakup.** Основной путь — GraphQL (лот+объявление+заказчик одним
   запросом), но клиент инкапсулирует протокол: если GraphQL-схема «поедет», переключимся на
   REST v3 (`/v3/trd-buy`, `/v3/lots`) без правок остального кода. Плюс третий режим —
   публичный HTML-поиск `/ru/search/announce` (работает без токена, проверено) как аварийный.
5. **Курсор на коннектор хранится в БД** (таблица `connector_state`: last_run_at, last_cursor,
   last_error), а не в файлах — это же кормит мониторинг коннекторов в дашборде (Этап 3).
6. **Реестр источников = config/sources.yaml** (создан, все источники проверены вживую
   05.07.2026). Два домена из брифа мертвы: qazstore.kz → skstore.kz, tender.bi.group →
   bnect.pro. e-sauda.kz оказался аукционом ИМУЩЕСТВА (не спрос) — отклонён. etbemp.kz,
   adata-тендеры, kazcontract — закрыты авторизацией/платные — отклонены.
7. **Единая модель повышена до «сигнал + воронка»**: статусы SignalStatus зашиты в модель и
   обязаны 1:1 совпадать с дашбордом и PLAYBOOK.md (Definition of Done этапов 3+).
8. **ИИ-слой получил фабрику и контракт деградации**: любой провайдер возвращает `None` при
   недоступности, вызывающий код обязан иметь эвристический путь. Это тестируется.

## 4. Архитектура (итог)

```
┌ connectors/ (1 источник = 1 модуль; mock→live по ключу в .env)
│    goszakup, mitwork, med_ecc, mpkz, samruk, ... → Iterable[DemandSignal]
├ net/        Fetcher (ретраи+backoff, троттлинг/Crawl-delay, UA-ротация, Retry-After)
│             ProxyProvider (NoProxy сейчас; пул прокси позже БЕЗ правки коннекторов)
├ core/       models (DemandSignal, статусы воронки), pipeline (нормализация→фильтр→
│             дедуп→классификация→обогащение→скоринг→маршрутизация), scheduler (Э7)
├ classify/   ключевые слова + префиксы ЕНС ТРУ (config/categories.yaml); LLM — опция
├ enrich/     контакты ЛПР: Statsnet/Kompra/Adata (mock) (Э2)
├ scoring/    бюджет + регион + свежесть + категория (Э2)
├ storage/    SQLite (schema versioned), repo-слой; connector_state = курсоры+health
├ llm/        LLMProvider: Null (дефолт) | Ollama (qwen3:8b) | ClaudeCode
├ notify/     Telegram-алерты + дневной дайджест; dry-run в консоль без токена
├ dashboard/  FastAPI + Jinja2: воронка, метрики, фильтры, статусы, мониторинг, отчёт (Э3)
└ cli         --once | --backfill N | --dry-run | --connector X | daemon (Э7)
```

## 5. Этапы и критерии приёмки

### Э0. Discovery + каркас — ТЕКУЩИЙ
Сделано: исследованы все источники (4 отчёта), снята фактическая GraphQL-схема goszakup,
git-репо + venv + pytest + ruff + CI, абстракции net/llm/connector/models, sources.yaml,
categories.yaml (черновик ТРУ/ОКЭД), PLAN/PROGRESS/ASSUMPTIONS/DECISIONS.
**Приёмка:** тесты зелёные; ruff чист; все артефакты в git; сводка улучшений показана владельцу.

### Э1. Вертикальный срез: goszakup mock→live
Ядро pipeline + storage(SQLite: signals, connector_state) + коннектор goszakup
(GraphQL-клиент; mock на фикстурах, повторяющих РЕАЛЬНУЮ форму ответа ows) + фильтр по
categories.yaml + дедуп + CLI `--once --dry-run --connector goszakup` + Telegram-нотификатор
(консольный dry-run без токена).
**Приёмка:** `python -m demandradar --once --dry-run` на фикстурах выводит нормализованные,
отфильтрованные, дедуплицированные сигналы; повторный запуск даёт 0 новых (дедуп работает);
тесты этапа зелёные; score ≥ 85.

### Э2. Обогащение + скоринг + закалка БД
Enrich-слой (Statsnet/Kompra mock: контакты ЛПР по БИН), скоринг (бюджет/регион/свежесть/
категория, веса в конфиге), миграции схемы, LLM-классификация «мутных» лотов как опция.
**Приёмка:** сигнал с БИН получает контакт (mock); топ-N по score корректен на тестовом
наборе; при DR_LLM_PROVIDER=null всё работает эвристиками.

### Э3. Дашборд + уведомления
FastAPI+Jinja2: воронка (new→in_work→contacted→kp_sent→meeting→deal/rejected), метрики
за день/неделю/всё время, разбивки по источникам/категориям/регионам/типам спроса, топ по
score, смена статуса из UI, мониторинг коннекторов (last_run, ошибки, жив/упал), кнопка
«Запросить отчёт» → CSV. Telegram: алерты по score-порогу + дневной дайджест. PLAYBOOK.md
(разделы 0-2, статусы = дашборду).
**Приёмка:** дашборд поднимается локально, воронка и отчёт работают на данных mock-прогона;
статус меняется из UI и сохраняется; экспорт CSV скачивается.

### Э4. Мультиконнектор формализованного спроса
mitwork (scrape, crawl-delay 10), med_ecc+fms_ecc (scrape), mpkz (scrape, категория мебели),
samruk (реверс JSON или headless — по результату разведки), сверка покрытия через tenderbot.
**Приёмка:** ≥3 новых коннекторов работают через тот же интерфейс БЕЗ правок ядра;
у каждого - фикстуры и тесты; мониторинг в дашборде показывает их статус.

### Э5. Опережающий спрос + Модуль B
Новые ТОО/ИП по ОКЭД+региону (Statsnet/data.egov, mock→live), стройсигналы (tizilim/bnect),
новости RSS-пула по ключам, hh.kz-вакансии; Модуль B: watchlist-профили компаний, карточка
«игрок→объект→фаза→сигналы→контакты», агрегация разнородных сигналов в профиль.
**Приёмка:** демо-кейс «новая сеть открывает магазин» собирается из ≥2 типов сигналов в одну
карточку watchlist; новые компании фильтруются по ОКЭД из categories.yaml.

### Э6. Латентный спрос + Модуль A + каналы действий
Wordstat (mock→live), satu/маркетплейсы (по итогам разведки), krisha-ассистент (ТОЧЕЧНЫЙ разбор
по URL, не массовый), RealEstateContact + реферальная связка «партнёр→лид→сделка→%» в дашборде,
экспорт очереди обзвона (CSV), модуль точечной рассылки (подтверждение + лог + отписка).
PLAYBOOK.md завершён + сборка .docx (pandoc, опенсорс).
**Приёмка:** латентный сигнал появляется в общей воронке; реферальная карточка живёт в дашборде;
рассылка требует явного подтверждения и логируется; PLAYBOOK синхронен с UI.

### Э7. 24/7 + эксплуатация
Демон (интервалы на коннектор из sources.yaml, продолжение с курсора, graceful shutdown),
health-check эндпоинт, Docker+compose, деплой-гайд VPS, полный тест-сьют, финальный README,
итоговый отчёт.
**Приёмка:** `docker compose up` поднимает демон+дашборд; kill -9 демона не теряет курсоры;
падение одного коннектора не роняет остальные (тест); критерий завершения брифа выполнен.
